# 2026-06-09 수정사항 — "가짜 연결" 재접속 무한루프 + TIME_WAIT 폭증 해결

## 증상 (현장 보고)

- 무전기 채널이 **연결됨 ↔ 끊김을 무한 반복** (로그상 항상 "Reconnecting... attempt 1").
- 시간이 지나면 양쪽 채널 다 끊김(검은 화면).
- `netstat` 상 `127.0.0.1:xxxxx → 127.0.0.1:8554`(viewer→MediaMTX) **TIME_WAIT 연결이 무한정 증가**.
- 채널(빈 채널 포함)을 추가하면 일시적으로 해결됨.
- 사이드바에 "연결됨"으로 뜨는데 `0.0 Mbps | 0.0 fps`.

```
19:21:57  Reconnecting... attempt 1
19:21:57  [RTSP] session ... reading from path 'radio20', with TCP, 1 track (H264)
19:21:57  Stream connected
19:22:07  VLC: no data received in 10s, aborting
19:22:07  Disconnected
19:22:12  Reconnecting... attempt 1   ← 또 attempt 1 (카운터가 리셋됨)
```

## 근본 원인

`libvlc_MediaPlayerPlaying` 이벤트는 **RTSP 세션이 열리고 트랙 정보(SDP)를 받은 시점**에
발동한다 — **실제 영상 프레임이 한 장도 안 왔어도** 발동한다.

MediaMTX는 `sourceOnDemand: false`로 원본(카메라)이 한 번 붙었을 때 받아둔 H264 트랙정보를
캐시한다. 그래서 카메라가 조용해져도 viewer가 접속하면 **세션을 열어주고 트랙정보만 응답**한다
(= "가짜 연결"). VLC는 이걸로 "Playing"을 발동.

기존 `onPlaying()`은 이 시점에:
- `m_reconnectCount = 0` (재접속 카운터 리셋)
- `m_hasConnectedOnce = true`
- 상태 = `Connected`

로 확정해버렸다. 그 뒤 실제 데이터가 없어 10초 후 VLC가 `no data received → abort` →
`Disconnected` → 재접속. 그런데 **다음 가짜 Playing이 또 카운터를 0으로 리셋**하므로:

- give-up 로직(`m_reconnectCount > maxAttempts`)이 **절대 발동 못 함** → 무한 재접속
- 매 사이클마다 새 TCP(127.0.0.1→8554) 생성 → **TIME_WAIT 무한 누적**

즉 v0.0.8의 give-up 안전장치가 "가짜 Playing의 카운터 리셋"에 의해 무력화되어 있었다.

> "빈 채널 추가 시 해결"의 이유: 채널 변경 → MediaMTX config hot-reload → 소스 재-dial →
> 잠깐 실제 프레임이 다시 흐름. 카메라가 다시 조용해지면 재발(임시방편).

## 변경 내역 (`src/VlcWidget.cpp`, `src/VlcWidget.h`)

핵심: **"RTSP 세션 열림(Playing)"과 "실제 프레임 수신"을 분리**한다.

### 1. `onPlaying` — 연결 확정 제거
- 더 이상 여기서 카운터 리셋/`Connected` 확정을 하지 않는다.
- 상태를 `Connecting`(수신 대기)로 두고, **데이터 워치독(5초)**을 무장한다.
- 로그: "RTSP 세션 열림 — 영상 데이터 대기"(DEBUG).

### 2. 실제 프레임 수신 시에만 연결 확정 — `confirmDataFlowing()`
- `refreshStats()`(1초 주기)에서 `displayed_pictures > 0 || decoded_video > 0` 감지 시 호출.
- 이때 비로소 `Connected` + `m_hasConnectedOnce=true` + `m_reconnectCount=0` + 워치독 정지 +
  "Stream connected" 로그.

### 3. 데이터 워치독 — `onDataWatchdogTimeout()`
- Playing 후 `DATA_CONFIRM_TIMEOUT_MS`(5초) 안에 실제 프레임이 안 오면 가짜 연결로 판단.
- 죽은 세션을 즉시 `cleanupPlayer()`로 정리(TIME_WAIT 체류 단축) 후 `scheduleReconnect()`.
- 이 재접속은 `tryReconnect()`에서 **카운터를 증가**시키므로 give-up으로 수렴한다.

### 4. give-up 횟수 (relay)
- `INITIAL_RELAY_MAX_RECONNECT`: **24 → 5** (사용자 지정 "5초 기준 5회").
- 즉 데이터가 안 오는 relay 경로는 `5초 × 5회 ≈ 25초` 후 "오프라인 - 수동 재연결 필요"로 멈춤.
- `INITIAL_DIRECT_MAX_RECONNECT = 3`(직접 RTSP), `ESTABLISHED_MAX_RECONNECT = 10`(연결됐던 채널)은 유지.

### 5. 중복 처리 방지
- `onError`/`onEndReached` 진입 시, `stop()`, give-up 분기, 새 세션 시작(`startPlaybackPlayer`)에서
  데이터 워치독을 정지/리셋.

## 효과

- 카메라 OFF/원본 끊김 → **세션 5회 시도(약 25초) 후 자동 포기** → 재접속 루프 종료, TIME_WAIT 누적 정지.
- 사이드바 "연결됨"은 **실제 영상이 흐를 때만** 표시(0fps 가짜 연결 표시 제거).
- 카메라 복귀 시: 수동 "재연결" 또는 채널 수정(hot-reload)으로 재개.

## 검증

- **빌드:** `cmake --build build` 성공(경고/에러 없음). 산출물 `build/viewer.app`.
- **현장 검증 필요:**
  1. 카메라 한쪽 OFF 시 해당 채널이 ~25초 후 "오프라인 - 수동 재연결 필요"로 멈추는지,
     `netstat`의 8554 TIME_WAIT가 더 이상 증가하지 않는지.
  2. 정상 카메라가 "연결중" → (수초 내) "연결됨"으로 바뀌고 영상이 나오는지.
  3. 카메라 복귀 후 우클릭 "재연결"로 정상 재생되는지.

## 튜닝 포인트

- 현장 카메라 부팅이 느려 25초 안에 못 올라오면 `INITIAL_RELAY_MAX_RECONNECT`(현재 5)를 키운다.
- 데이터 확정 대기시간은 `DATA_CONFIRM_TIMEOUT_MS`(현재 5000ms).

## 변경 파일

- `src/VlcWidget.h` — `m_dataConfirmed`, `m_dataWatchdogTimer`, `DATA_CONFIRM_TIMEOUT_MS`,
  `INITIAL_RELAY_MAX_RECONNECT 24→5`, `onDataWatchdogTimeout()`/`confirmDataFlowing()` 선언.
- `src/VlcWidget.cpp` — 워치독 타이머 생성, `onPlaying` 확정 제거, `refreshStats` 데이터 확정 연결,
  `confirmDataFlowing()`/`onDataWatchdogTimeout()` 구현, `stop()`/give-up/onError·onEndReached 정리.

# 2026-06-09 (이어서) — 끊김의 진짜 원인: VLC↔MediaMTX TCP 인터리빙 + UDP 전환

위 재접속 로직 수정 후에도 현장에서 "연결됨→끊김→재시도"가 반복 보고됨. VOXL 벤치
(ADB 연결, Mac eth0 `169.254.4.2` ↔ VOXL `169.254.4.1` 직결)에서 레그별로 분리 측정.

## 측정 결과 (레그 분리)

| 레그 | 결과 | 증거 |
|---|---|---|
| 카메라 직결 (Mac→voxl-streamer, TCP) | ✅ 안정 | ffmpeg 94.8초 연속, 2838프레임, 에러 0. GOP≈1초(즉시 I-frame) |
| 카메라→MediaMTX (sourceOnDemand:false) | ✅ 안정 | API: `ready/online=true`, `bytesReceived` 단조 증가(40MB+) |
| **MediaMTX→VLC (로컬, TCP 인터리브)** | ❌ 간헐 끊김 | `received unexpected interleaved frame`, ~65초 후 드롭 |

소스(UVC 640x480x30 → voxl-streamer h264 1Mbps)와 카메라 레그는 완전히 안정적.
문제는 **VLC↔MediaMTX 로컬 레그의 RTSP-over-TCP 인터리빙**에서만 발생.

## 결정적 재현

별도 MediaMTX(`rtspTransports: [udp, tcp]`)를 띄우고 VLC로 같은 소스를 끌어봄:

```
session a0b001a7  is reading ... with UDP   → 정상 스트리밍
session fe9dce52  is reading ... with TCP
closed: received unexpected interleaved frame   ← TCP 세션은 2초 만에 끊김
```

동일 VLC·MediaMTX에서 **UDP 세션은 정상, TCP 세션은 즉시 `unexpected interleaved frame`**.
= VLC(libVLC 3.0.23) ↔ MediaMTX v1.19.0(gortsplib) 간 RTSP-over-TCP 인터리빙 비호환.

## 수정 (P0)

로컬 레그(VLC↔MediaMTX, 항상 127.0.0.1)를 **UDP**로 전환. loopback UDP는 무손실이며
인터리빙 메커니즘 자체가 없어 이 실패 모드가 발생 불가능.

- `MediaRelayManager.cpp` `buildConfig()`: `rtspTransports: [tcp]` → **`[udp, tcp]`**
  (UDP 수락 + TCP fallback 유지). path별 `rtspTransport: tcp`(카메라 레그)는 **유지**.
- `VlcWidget.cpp` `openPlaybackMedia()`: 로컬 URL의 **`:rtsp-tcp` 제거** → VLC 순정 UDP 협상.
- `openRecordingMedia()`: **녹화는 TCP 유지**(파일 무결성 우선, 사용자 결정).

## 검증

- 새 설정 MediaMTX + ffmpeg UDP pull: 637프레임/21초 연속, 에러 0, `with UDP` 확인.
- VLC UDP 세션 정상 / TCP 세션은 2초 내 `unexpected interleaved frame` 재현(대조군).
- 빌드 성공(경고/에러 없음).

## 현장 검증 필요 (실배포 = Windows)

1. 새 빌드로 viewer 10분+ 구동 → 로그에서 `unexpected interleaved frame` /
   `5초간 영상 데이터 없음` / `Disconnected` 가 사라지는지.
2. `netstat` 8554 TIME_WAIT 누적 멈춤(UDP라 TCP 소켓 자체가 거의 안 생김).

## 주의 / 후속

- VLC는 UDP 우선 협상하나, UDP RTP 미수신 시 **TCP fallback** 가능성 있음. loopback에선
  UDP가 실패할 일이 거의 없지만, fallback 시 다시 이 버그에 노출될 수 있음. 필요하면
  강제 UDP 검토. (현재는 데이터 워치독 5초가 2차 안전망으로 동작)
- 근본적으론 VLC↔MediaMTX TCP 인터리빙 비호환을 MediaMTX/VLC 버전 업으로 추적 가치 있음.

# 2026-06-09 (정정) — UDP 전환 철회, TCP 유지

위 "UDP 전환(P0)"은 **철회**한다. 추가 검증에서 다음이 드러남:

- 로컬 레그를 UDP 로 바꾸면 `unexpected interleaved frame` 은 사라지지만,
  **libVLC 가 UDP RTP 를 수신하지 못한다.** (viewer 임베드 / VLC CLI 둘 다 미수신,
  ffmpeg 는 동일 경로 UDP 정상 수신 → libVLC 한정 이슈. macOS 벤치 기준.)
- 증상: MediaMTX `bytesSent` 는 증가(보내고는 있음)하지만 VLC 통계는 전부 0
  (`read_bytes`/`input_bitrate`/`decoded_video` = 0), 세션 `bytesReceived`(RTCP) 정체.
- 앞선 "UDP 90초 안정"은 MediaMTX `bytesSent`(보낸 양)만 보고 판단한 오판이었음.
  실제 수신은 VLC `read_bytes`/디코드로 확인해야 한다.

## 결론

- **로컬 레그(VLC↔MediaMTX)는 TCP 유지** (libVLC 정상 수신).
  - `MediaRelayManager.cpp`: `rtspTransports: [tcp]` (원복).
  - `VlcWidget.cpp openPlaybackMedia()`: 로컬 URL 에 `:rtsp-tcp` (원복).
- TCP 인터리빙 간헐 끊김(`unexpected interleaved frame`)은 **재접속/데이터워치독으로 자동 복구**.
  (무한루프/TIME_WAIT 폭증은 이미 해결됨 — 위 첫 섹션.)
- 검증 팁: TCP 는 신뢰전송이라 MediaMTX `bytesSent` 가 꾸준히 늘면 backpressure 상
  VLC 가 실제로 읽는 것(=수신). UDP 는 `bytesSent` 가 늘어도 VLC 미수신일 수 있음.

## 남은 과제 (선택)

- TCP 인터리빙 비호환 자체를 없애려면 MediaMTX 버전업/옵션 또는 libVLC 옵션 조사.
- libVLC UDP 미수신의 근본 원인(포트 바인딩/플랫폼)은 Windows 에선 다를 수 있음 — 필요 시 별도 확인.
