# 2026-06-06 수정사항 — 영상 지연 누적 문제 해결

## 배경

영상을 오래 재생할수록 지연(latency)이 단조 누적되고, 앱을 종료/재실행해야만 정상으로
돌아오는 문제가 보고되었다. 로그 분석 결과 다음 패턴이 확인됨:

```
[ERR] VLC: ES_OUT_SET_(GROUP_)PCR is called too late (jitter of 72ms → 381ms → 750ms ... ignored)
[ERR] VLC: more than 5 seconds of late video -> dropping frame (computer too slow?)
```

### 근본 원인

`src/main.cpp`의 VLC 인스턴스 옵션 `--clock-jitter=0` + `--clock-synchro=0` 조합이
**VLC의 클럭 재동기화·지터 보상 기능을 꺼버린 것**이 원인이었다. (VLC 3.0.23 실측 기준)

| 옵션 | `=0`의 실제 의미 | 결과 |
|---|---|---|
| `--clock-jitter=0` | 동기화가 보상할 최대 입력 지연 = 0ms | 정상 RTSP 지터를 전혀 보정하지 않고 PCR 갱신을 무시 |
| `--clock-synchro=0` | 클럭 동기화 OFF (`-1`이 기본/auto, `0`은 비활성) | 어긋난 클럭을 재동기화하지 않음 |

카메라와 PC 클럭의 미세 드리프트가 보정 없이 계속 쌓여 지연이 단조 누적 → 5초 초과 시
프레임 드롭 → 재시작 전까지 고착. 재시작 시 클럭 기준이 재초기화되어 정상 복귀했던 것.

> 참고: "libvlc에는 리셋 알고리즘이 없다"는 통설은 사실이 아니다. libvlc와 VLC 플레이어는
> 동일 엔진을 쓰며, 이 앱이 위 옵션으로 그 기능을 직접 꺼놨던 것뿐이다.

## 변경 내역

### 1. WARN 로그 삭제
- **파일:** `src/main.cpp` (`vlcLogCallback`)
- **변경:** 로그 등급 필터를 `LIBVLC_WARNING` → `LIBVLC_ERROR`로 상향.
- **효과:** 경고성 노이즈(surface dimensions, PCR jitter 등)가 사라지고 ERROR만 표시.

### 2. TCP → UDP 연결
- **파일:** `src/VlcWidget.cpp` (`openPlaybackMedia`)
- **변경:** 재생 미디어에서 `:rtsp-tcp` 옵션 제거 (VLC 순정 기본값이 UDP).
- **효과:** 음영지역에서 끊겼다 붙을 때 TCP 재핸드셰이크/재전송 지연 없이 최신 패킷부터
  즉시 수신 → 화면 복구 시간 단축.
- **트레이드오프:** 손실 구간에서 깨짐(매크로블로킹) 증가 가능 → 캐시 상향(5번)으로 완충.
- **주의:** **녹화 경로(`openRecordingMedia`)는 `:rtsp-tcp` 유지** (저장 파일 무결성 우선).
  녹화도 UDP로 통일하려면 해당 옵션 한 줄만 제거.

### 3. 지터 관련 옵션 삭제 (VLC 순정화)
- **파일:** `src/main.cpp` (인스턴스 args)
- **변경:** `--clock-jitter=0`, `--clock-synchro=0`, `--avcodec-hurry-up`, `--avcodec-fast` 제거.
- **효과:** VLC 순정 엔진의 클럭 재동기화·지터 보상이 부활. 일시적으로 밀려도 엔진이 스스로
  따라잡아 지연이 0 근처로 회복됨. (= 지연 누적 문제의 근본 해결)

### 4. 오디오 미사용 유지
- **변경 없음.** 기존 `--no-audio` / `:no-audio` / `:audio-track=-1` 유지.
- **부가 효과:** 오디오 트랙이 비디오 클럭(PCR)을 끌고 가는 동기화 꼬임 가능성 배제.

### 5. 캐시 500ms → 1000ms
- **파일:** `src/main.cpp` (인스턴스), `src/VlcWidget.cpp` (per-media)
- **변경:** `network-caching` / `live-caching` 를 양쪽 모두 1000으로 상향.
- **효과:** 무선 지터(링크 불량 시 500ms↑) 흡수 버퍼 확보 → 깨짐/끊김 감소.
- **트레이드오프:** 기저 지연 최대 약 +0.5초 (실시간 관제 허용 범위).

### 6. 지연 누적 워치독 추가 (2차 안전망)
- **파일:** `src/VlcWidget.cpp` (`maybeFlushForLatency`), `src/VlcWidget.h` (상수/멤버)
- **목적:** 순정화 후에도 RF 하드 드롭 등으로 클럭이 드물게 고착될 때, 이를 자동 감지해
  스트림을 flush(재접속)시켜 회복.

#### 동작 흐름 (1초 주기 통계 갱신 `refreshStats`에 연동)
```
매 1초마다:
  연결(Connected) 상태가 아니면 → 카운터 0 리셋, 종료
  지난 1초간 "늦어서 버린 프레임"(lost_pictures)이 있었나?
    없으면 → 카운터 0 리셋 (정상)
    있으면 → 카운터 +1
  카운터가 10 도달 + 마지막 flush로부터 60초 경과 시:
    → 로그 남기고 reconnect() 실행 (스트림 flush)
    → 카운터/타이머 리셋
```

#### 설정값
| 항목 | 값 | 의미 |
|---|---|---|
| 감지 신호 | `lost_pictures` 증가 | 늦어서 버린 프레임 = 밀림 고착 증거. (UDP는 discontinuity가 정상적으로도 잦아 미사용) |
| `WATCHDOG_LATE_STREAK` | 10 | 10초 *연속* 드롭 시 발동 (오발동 방지) |
| `WATCHDOG_MIN_INTERVAL_MS` | 60000 (60초) | flush 후 1분간 재발동 억제 (재접속 루프 방지) |
| flush 동작 | `reconnect()` | player 정리 후 재생 → 클럭 기준 재초기화 (재시작과 동일 회복) |
| 실행 시점 | `QTimer::singleShot(0, ...)` | 통계 루프 밖에서 비동기 실행 → player 교체 블로킹이 UI 멈춤 방지 |

#### 동작 예시
- 정상(드롭 0): 카운터 계속 0 → 발동 안 함.
- 잠깐 깨짐(3초 드롭 후 회복): 카운터 3 → 0 리셋 → 발동 안 함.
- 하드 고착(10초+ 연속 드롭): 10초째 발동 → 자동 재접속 회복, 이후 60초 억제.

> 임계값(10초/60초)은 현장 무선 상태에 맞게 조정 가능. 민감하면 숫자를 키우고, 회복이
> 굼뜨면 10초를 줄인다.

## 검증

- **빌드:** `cmake --build build` 성공 (경고/에러 없음). 산출물 `build/viewer.app`.
- **현장 검증 필요 (실제 무선 카메라):**
  1. 수십 분~수 시간 재생 시 지연이 0 근처 유지 (이전: 단조 증가)
  2. 음영지역 진입→복구 시 화면 복구 시간 (UDP 효과)
  3. 손실 구간 깨짐 정도 (과하면 TCP 복귀 또는 캐시 추가 상향 검토)
  4. 로그 탭에 WARN 사라지고 ERROR만 표시되는지
